# Диагностический отчёт: Каскадное удаление иерархии заметок

**Дата:** 24 мая 2026  
**Серьёзность:** MEDIUM  
**Статус:** Диагностика завершена  
**Баг воспроизведён:** ДА

---

## Краткое описание бага

При удалении родительской заметки через `DELETE /api/notes/{id}` все дочерние заметки (с `parentId` указывающим на удаляемую заметку) каскадно удаляются из базы данных вместо того, чтобы быть перемещёнными на корневой уровень. Это приводит к безвозвратной потере данных.

**ВАЖНО:** Исходное описание бага упоминает `FolderService.deleteFolder()`, но этот класс **НЕ СУЩЕСТВУЕТ** в кодовой базе. Приложение использует иерархию заметок через поле `parentId` в `NoteService`, а не отдельную концепцию папок.

---

## Корневая причина

**Файл:** `server/src/main/kotlin/com/notebox/domain/note/NoteController.kt`  
**Строка:** 90  
**Уверенность:** ВЫСОКАЯ

```kotlin
@RequestParam(required = false, defaultValue = "cascade") action: String
```

API endpoint по умолчанию использует каскадное удаление. Это передаётся в `NoteService.deleteNote()`, который вызывает `NoteRepository.deleteWithDescendants()`, явно удаляющий все дочерние заметки.

### Цепочка вызовов:

| Уровень | Файл:Строка | Проблема |
|---------|-------------|----------|
| API | `NoteController.kt:90` | `defaultValue = "cascade"` |
| Service | `NoteService.kt:149` | `cascadeDelete: Boolean = true` |
| Repository | `NoteRepository.kt:247-254` | `deleteWithDescendants()` удаляет всё |

### Противоречие с базой данных:

Миграция `V001__add_parent_id_to_notes.sql` определяет:
```sql
ALTER TABLE notes ADD CONSTRAINT fk_notes_parent
    FOREIGN KEY (parent_id) REFERENCES notes(id) ON DELETE SET NULL;
```

База данных правильно настроена на `SET NULL` (осиротить детей), но код приложения **обходит** это ограничение, явно удаляя все потомки.

## Находки субагентов

### Architect (Трассировщик кода)
- **Согласие с корневой причиной:** ДА
- **Ключевые находки:**
  - Корневая причина в `NoteController.kt:90` - defaultValue = "cascade"
  - Метод `deleteWithDescendants()` использует рекурсивный CTE для сбора всех потомков
  - Баг существует с момента добавления функции вложенных страниц (commit c581b60)
  - Существует альтернативный метод `orphanChildren()`, но он не вызывается по умолчанию

### Stack Expert (Эксперт по технологиям)
- **Согласие с корневой причиной:** ДА
- **Ключевые находки:**
  - Exposed ORM 0.47.0 не имеет CASCADE constraints в определении таблицы
  - PostgreSQL 15 полностью поддерживает `ON DELETE SET NULL`
  - Ограничение БД корректно настроено, но обходится кодом приложения
  - Spring Boot 3.2.2, Kotlin 1.9.22 - стабильные версии

### Консенсус
Оба субагента полностью согласны:
- Проблема в **логике приложения**, а не в конфигурации ORM или БД
- Корневая причина - дефолтное значение `"cascade"` в API endpoint
- Исправление требует изменения поведения по умолчанию
