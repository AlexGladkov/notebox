# Диагностический отчёт: Background polling hits wrong API URL causing 403 errors

**Дата:** 2026-05-22  
**Серьёзность:** MEDIUM  
**Статус:** КОРНЕВАЯ ПРИЧИНА ОПРЕДЕЛЕНА  
**Уверенность:** HIGH

---

## Краткое описание бага

Фронтенд делает периодические фоновые запросы к `/api/notes` вместо `/notebox/api/notes` каждые 30 секунд. Это вызывает ошибки 403/401 в консоли браузера. Причина — жёстко закодированный путь API в сервисе проверки сетевого подключения, который игнорирует централизованную конфигурацию `API_BASE_URL`.

---

## Корневая причина

### Основной файл с ошибкой

**Файл:** `src/services/offline/networkStatus.ts`  
**Строка:** 11  
**Функция:** `checkConnectivity()`

```typescript
// ОШИБКА - жёстко закодированный путь, игнорирует API_BASE_URL
const response = await fetch('/api/notes', {
  method: 'HEAD',
  cache: 'no-cache',
  credentials: 'include',
});
```

### Почему это баг

1. **Жёстко закодированный endpoint:** Сервис проверки сетевого статуса делает прямой fetch к `/api/notes`
2. **Игнорирует централизованную конфигурацию:** В отличие от других API-вызовов, использующих `apiClient.get()`, этот запрос не использует `API_BASE_URL`
3. **Не поддерживает path prefix:** При деплое с префиксом пути (например, `/notebox`):
   - Запрос идёт к: `http://host:33590/api/notes` (неверно)
   - Должен идти к: `http://host:33590/notebox/api/notes` (верно)
4. **Периодический сбой:** Ошибка повторяется каждые 30 секунд из-за `setInterval` на строке 38

### Архитектурная проблема

Кодовая база использует два паттерна для API-запросов:

**Паттерн A (Правильный) — используется в `api/client.ts`:**
```typescript
const API_BASE_URL = import.meta.env.VITE_API_URL ?? '';
const response = await fetch(`${API_BASE_URL}${path}`, { ... });
```

**Паттерн B (Неправильный) — используется в `networkStatus.ts`:**
```typescript
const response = await fetch('/api/notes', { ... }); // Игнорирует API_BASE_URL
```

## Результаты Consilium-анализа

### Архитектор (Code Tracer)
- **Согласие по корневой причине:** ДА
- **Выводы:** Проблема в `networkStatus.ts:11`. Сервис был добавлен в рамках функции offline-синхронизации без интеграции с существующей системой конфигурации API. Коммит c20fa74 исправил API-конфигурацию для большинства файлов, но пропустил `networkStatus.ts`.

### Stack Expert (Фреймворк-анализ)
- **Согласие по корневой причине:** ДА
- **Выводы:** Пустая переменная `VITE_API_URL` в `.env.production` усугубляет проблему. Vite `base` настроен как `/`, что не учитывает деплой на `/notebox/`. Конфигурация Nginx не обрабатывает `/notebox` prefix.

### Консенсус
**Полное согласие** по корневой причине: жёстко закодированный URL в `networkStatus.ts:11` игнорирует централизованную API-конфигурацию.

## План исправления

### Шаг 1: Добавить импорт API_BASE_URL в networkStatus.ts

**Файл:** `src/services/offline/networkStatus.ts`

Добавить в начало файла:
```typescript
const API_BASE_URL = import.meta.env.VITE_API_URL ?? '';
```

### Шаг 2: Исправить функцию checkConnectivity()

**Файл:** `src/services/offline/networkStatus.ts`  
**Строка:** 11

**До:**
```typescript
const response = await fetch('/api/notes', {
```

**После:**
```typescript
const response = await fetch(`${API_BASE_URL}/api/notes`, {
```

### Альтернативное решение (предпочтительнее)

Использовать централизованный `apiClient` вместо raw fetch:

```typescript
import { apiClient } from '../../api/client';

async function checkConnectivity(): Promise<boolean> {
  try {
    await apiClient.get('/api/notes');
    return true;
  } catch (error) {
    // Проверяем, что это не сетевая ошибка
    if (error instanceof Error && error.name === 'ApiError') {
      // Сервер ответил (даже с ошибкой авторизации) — сеть работает
      return true;
    }
    return false;
  }
}
```

## Заключение

Это **баг консистентности конфигурации**, где один сервисный модуль (`networkStatus`) был добавлен без соблюдения установленного паттерна конфигурации API. Исправление требует либо:

1. Рефакторинг `networkStatus.ts` для использования централизованного `apiClient`, ИЛИ
2. Добавление переменной `API_BASE_URL` из environment-конфигурации

Баг проявляется как ошибки 403/401 каждые 30 секунд в консоли браузера при деплое за reverse proxy с path prefix типа `/notebox`.
